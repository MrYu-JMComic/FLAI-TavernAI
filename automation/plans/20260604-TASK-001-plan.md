## 优先级

## 详细描述

### 背景
`backend/src/services/providers.js` 是项目中最复杂的后端模块（~950 行），负责：
- Provider 配置规范化 (`normalizeProviderRow`, `providerWithSecret`)
- API Key 加密/解密状态管理 (`resolveApiKey`)
- 请求体构建 (`buildProviderBody`)
- 非流式补全 (`generateCompletion`, `runToolCompletion`)
- 流式补全 (`streamCompletion`, `streamToolCompletion`)
- DeepSeek 余额查询 (`fetchDeepSeekBalance`)
- 模型列表获取 (`listProviderModels`)
- Usage 快照与费用计算 (`buildUsageSnapshot`, `summarizeUsageSnapshots`)

当前**零测试覆盖**。该模块处理 API Key 加密、外部 HTTP 请求、SSE 流解析、错误重试等关键路径，任何回归都直接影响用户核心体验。

### 任务范围
编写 `backend/src/tests/providers.test.js`，覆盖以下场景：

#### 1. 纯函数测试（无需 mock）
- `normalizeProviderRow(null)` → 返回默认 DeepSeek 配置
- `normalizeProviderRow(row)` → 正确映射字段
- `providerWithSecret(row)` → 包含解密后的 apiKey
- `buildProviderBody(settings, messages, stream)` → 正确构建请求体
- `buildProviderBody` 带 options（temperature, maxTokens, topP, etc.）
- `buildProviderBody` DeepSeek provider 特殊字段（thinking, stream_options）
- `buildUsageSnapshot(null)` → null
- `buildUsageSnapshot(usage)` → 正确计算 token 和费用
- `buildUsageSnapshot` DeepSeek 不同模型定价
- `summarizeUsageSnapshots([])` → 零值
- `summarizeUsageSnapshots` 多个 usage 汇总
- `hasUsableProvider` 各种边界条件

#### 2. 需要 mock fetch 的测试
- `listProviderModels` baseUrl 缺失 → 抛错
- `listProviderModels` apiKeyError → 抛错
- `listProviderModels` apiKey 缺失 → 抛错
- `listProviderModels` 正常返回 → 解析模型列表
- `listProviderModels` 返回格式变异（data 数组 vs models 数组）
- `generateCompletion` 无可用 provider → 走 mock 路径
- `generateCompletion` 正常响应 → 返回 content/reasoning/usage
- `generateCompletion` HTTP 错误 → 抛出错误信息
- `streamCompletion` 无可用 provider → 走 mock 路径
- `streamCompletion` 正常 SSE 流 → emit content/reasoning 事件
- `streamCompletion` HTTP 错误 → 抛出错误信息
- `streamCompletion` SSE 中 [DONE] 信号 → 正常结束
- `streamCompletion` signal abort → 中断流
- `fetchDeepSeekBalance` apiKeyError → 抛错
- `fetchDeepSeekBalance` 正常响应 → 返回余额数据

### 设计决策
- 遵循 `backend.test.js` 现有模式：`node:test` + `node:assert/strict`
- 纯函数直接测试，HTTP 相关函数通过全局 `fetch` mock 测试
- 不引入额外依赖（如 sinon），使用 Node.js 内置 mock 能力
- 测试文件独立，可单独运行
